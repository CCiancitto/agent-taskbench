# Task Spec Guidelines

A good task spec is narrow, testable, and explicit.

## Include

- one clear objective
- allowed files or modules
- disallowed behavior
- required output format
- validation steps
- edge cases that matter

## Prefer

- concrete language over vague goals
- a small number of acceptance criteria
- examples of expected input and output
- local validation that does not require paid services

## Avoid

- multi-goal tasks that blur success criteria
- hidden dependencies
- ambiguous "improve everything" wording
- tasks that depend on unavailable external systems
